The Problem

Labs across Sweden were losing COVID sequencing data because the submission platform had no error handling — and had been built strictly to a happy path that didn't reflect how labs actually worked.

Background

An MVP built under urgency, now silently failing the labs that depended on it

GENSAM was a national platform for lab technicians to register, upload, and track COVID genomic sequencing data. The system played a role in understanding mutation trends and supporting pandemic response.

The platform had been built under urgency as an MVP. On the happy path, it worked. Anything off it — a failed submission, a format error, a partial upload — either failed silently or left users with no way to recover.

The result: data loss, duplicated manual effort, and labs that had stopped trusting the system entirely.

My Role

Solo designer running research and design end-to-end on a constrained government tech stack

UX/UI Designer at LH+P. I ran research and discovery, led design from wireframes through to high-fidelity handoff, and worked directly with the engineering team building on a constrained government tech stack. Same embedded team as SVEBar — I ran both projects in parallel.

Working on two concurrent projects with the same agency team meant deep familiarity with the stakeholder context and institutional constraints, but required rigorous prioritisation of time between projects.

Discovery

Interviews and working sessions across multiple regions — mapping the full deposition workflow from sample to confirmed upload

I ran interviews and working sessions with lab technicians across multiple Swedish regions. The goal was to understand the complete deposition workflow: from receiving a positive sample, to preparing sequencing files, to submitting and confirming successful upload.

1
The MVP had no off-ramp from the happy path

No error states, no status tracking, no way to correct or update a submission once started. Users had built workarounds: spreadsheets to track batches, phone calls to confirm receipt, manual re-submissions when in doubt.

2
The trust gap was the core problem

Several labs had stopped actively using the platform and were emailing files directly instead. Solving the UX wasn't enough — the redesign needed to rebuild confidence in the system's reliability.

3
Test ID matching was a missing workflow step

Linking a sequencing result back to the original patient sample required a separate external spreadsheet. This was a critical workflow step the platform didn't support at all.

Baseline going in

Labs were actively working around the system. The clearest signal was the number of submissions that arrived via email rather than through the platform — a direct measure of lost trust. Reducing that was the redesign target.

The Hard Part

A government stack that couldn't build everything users needed, and a trust problem requiring technical fixes design couldn't fully provide

Government tech constraints

Some error handling that users genuinely needed wasn't buildable in the current funding round. Prioritisation calls had to be made and communicated clearly — to users and to the agency.

Designing for a system users had already abandoned

A cleaner UI wasn't going to fix the trust gap on its own. The design had to change the underlying experience of certainty: "I know what happened to my submission."

Scope uncertainty

The full feature scope wasn't clear at the start. Some features needed stakeholder discussion before they could be scoped. I had to prioritise design sequencing without knowing the full picture.

Honest expectation management

Users had legitimate complaints about backend reliability. Design could improve the experience, but restoring trust required being honest with both users and the team about what design alone could achieve.

How I Navigated It

The core design decision: make the user's progress visible instead of exposing system state

The insight that unlocked the design: instead of building better error states (which required significant backend work), I proposed a timeline-based mental model as the primary architecture. Making the user's journey visible was buildable within the current constraints — and it directly addressed the core anxiety: "did my submission actually go through?"

For Test ID matching, I proposed integrating it directly into the platform flow, eliminating the external spreadsheet dependency and removing a major failure point from the workflow.

For features that couldn't be built yet, I documented them in the product backlog with user research rationale — so the agency had a clear brief for the next funding round rather than a wishlist with no supporting evidence.

How Might We

Give lab technicians unambiguous certainty that their submission landed — within the constraints of a government tech stack that couldn't yet support full error handling?

Design

Timeline-based submission flow, real error states for the first time, and Test ID matching built into the platform

With the mental model reframed around submission progress rather than system state, I moved from discovery into design — building the timeline view, error handling, and test ID matching as a cohesive system rather than isolated improvements.

1
Timeline-based deposition overview

Showing the real-time status of each submission at a glance. The first thing users see is progress — not system state or error codes. Addresses the "did anything happen at all" anxiety directly.

2
Error states with visibility and recovery paths

The first real error handling the platform had. Each error is visible with specific guidance for next steps — not just "something went wrong."

3
Integrated Test ID matching

Eliminated the external spreadsheet dependency by building the matching workflow directly into the platform. Removed a major failure point and a significant source of manual effort.

4
Submission correction flow

Users can update or re-submit without starting over — the first time the platform supported any kind of recovery from a failed or partial submission.

5
Design system for future development

Built to public sector standards and extended with new patterns for the timeline view and error state hierarchy — both documented with rationale for the agency's engineering team.

Testing & Iterations

Low-fidelity walkthroughs targeting the three highest-anxiety moments in the submission workflow

Low-fidelity wireframes tested with 4 lab technicians across two sessions. Scenarios focused on the three highest-anxiety moments: submitting a batch, encountering an error mid-submission, and checking the status of a previous deposition.

Key validation: The timeline model was immediately intuitive. All four participants understood their position in the process without being prompted. Error visibility was described as "finally knowing what went wrong" by one participant who had previously experienced silent failures.

Findings informed two rounds of iteration before moving to high-fidelity: error state copy was rewritten to be more actionable, and timeline detail level was adjusted to match what users actually needed to see versus what felt like noise.

Results

Labs returned to the platform — and the email workaround that measured lost trust was eliminated

Rebuilt platform trust — labs that had stopped using GENSAM re-engaged after the redesign
First-ever error handling — submission failures now have a path forward, not a dead end
0 spreadsheets needed for Test ID matching — integrated directly into the platform flow
Full Sweden nationwide sequencing capacity fully supported across all regions

What I'd do differently

Argue harder for a proper error handling framework before the redesign started. We improved the UX around a system that still had technical gaps. Closing those gaps first would have made the design work more durable and the trust rebuild more complete. I flagged it — but I'd make the case more formally next time, with user research findings as the business case.

Design System

New patterns for the timeline view and error state hierarchy — documented for agency engineering teams to build on independently

Built to Swedish public sector design standards. Extended with new patterns for the timeline view and error state hierarchy — both documented with rationale and handed off to the agency engineering team for future development.

The timeline pattern and error state hierarchy were genuinely novel — the standards didn't cover them. Documentation included usage guidance, edge cases, and rationale so the team could extend them appropriately without requiring a designer's involvement for every decision.
